- **核心目标**：构建全球首个"智能路由降本 + 自建模型托管 + Web3透明对账"三位一体的AI推理聚合网关，帮助开发者及企业将AI推理成本降低40-60%。

## 1. 产品目标（3个清晰、正交的目标）

### 目标1：智能路由降本
**目标描述**：通过自研Judge Agent智能判断请求复杂度，将60%+简单请求路由至低成本自建DeepSeek-V4集群，实现整体推理成本降低40-60%。

**可量化指标**：
- 路由准确率：3个月85% → 6个月92% → 12个月95%
- 简单请求自处理率：≥60%
- P50延迟：3个月500ms → 6个月300ms → 12个月200ms

**正交性保证**：此目标专注于"智能判断与路由决策"，不涉及计费层或后端部署决策。

### 目标2：自建DeepSeek-V4推理集群
**目标描述**：部署高性能自建推理集群（vLLM/SGLang + MATDO优化），提供可与企业级商业API（如DeepSeek/Azure/Anthropic）相媲美但成本低20%+的服务。

**可量化指标**：
- 集群可用性：≥99.5%
- 故障自动切换时间：< 30秒
- 自建层吞吐：日均处理≥200M tokens

**正交性保证**：此目标专注于"基础设施与性能优化"，不涉及路由策略或计费逻辑。

### 目标3：Web3信用币透明对账
**目标描述**：引入$CRED代币体系，实现链下（Redis+PostgreSQL）+ 链上（L2）双层计费架构，提供实时成本可见性、多级预算控制、团队分账等财务级能力。

**可量化指标**：
- 计费准确率：≥99.99%
- 链上结算延迟：< 24小时（L2）
- $CRED持有者折扣：7折

**正交性保证**：此目标专注于"计费、对账与代币经济"，不涉及路由或推理逻辑。

## 2. 用户故事（3-5个场景）

### 场景1：AI应用开发者多模型快速接入
**角色**：张明，AI应用开发者，独立开发聊天机器人产品
**背景**：产品需要支持多种LLM能力（写作、代码、对话），目前直接调用各厂商API，SDK不一致且成本不可控
**需求**：
- 希望0代码改动迁移，只需改一行endpoint地址
- 希望一个SDK支持所有模型，无需管理多个密钥
- 希望看到实时成本看板，了解每种模型的花费
**成功标准**：迁移时间 < 30分钟；上线首周成本降低 ≥30%

### 场景2：SaaS企业团队成本分账
**角色**：李华，中小SaaS企业技术负责人，管理20人开发团队
**背景**：企业按项目使用AI能力，需要精确到团队/项目的成本分摊与预算控制
**需求**：
- 按团队、项目设置独立预算上限
- 每月生成财务级报表，支持导出
- 多人共享API Key但权限隔离
**成功标准**：月末对账时间从3天缩短到 < 1小时；财务部门可自助查看

### 场景3：跨境Web3项目方合规透明计费
**角色**：Sarah，Web3 DApp创始人，项目使用AI生成NFT描述和市场分析
**背景**：Web3项目方需要加密货币友好、账目公开可验证的计费方案
**需求**：
- 支持USDT/$CRED代币支付
- 所有交易链上可查，防止暗箱操作
- 需要合规的内容审核（NFT内容过滤）
**成功标准**：链上交易可验证；合规审核通过率 > 99%

### 场景4：成本敏感初创团队渐进式接入
**角色**：王强，初创AI公司CTO，考虑从OpenRouter迁移
**背景**：希望先低成本试用，逐步将核心业务迁移到平台
**需求**：
- 先用路由层（不部署自有模型），降低迁移风险
- 根据业务增长，渐进式使用自建推理层
- 随时可切换回原供应商
**成功标准**：试用转化率 > 60%；6个月内核心业务迁移率 > 80%

### 场景5：故障时自动兜底不宕机
**角色**：赵琳，企业DevOps工程师，关注服务连续性
**背景**：曾因单一API供应商故障导致服务中断24小时
**需求**：
- 主供应商故障时，< 30秒自动切换到备用供应商
- 切换过程对应用透明，无需改代码
- 故障原因和切换记录可追溯
**成功标准**：月度 SLA > 99.5%；故障期间用户体验无感知

## 3. 用户研究洞察（来自瑞思）

### 3.1 用户画像

| 用户类型 | 占比 | 核心诉求 | 支付意愿 |
|---------|------|---------|---------|
| AI应用开发者与初创团队 | 55% | 成本优化、多模型备份、快速接入 | $50-500/月 |
| 中小SaaS企业 | 30% | 透明度、团队分账、财务级报表 | $500-5000/月 |
| 跨境企业/Web3项目方 | 15% | 合规透明、加密货币友好 | $1000+/月 |

**关键发现**：成本敏感度最高（60%优先级），其次是透明度需求（40%），再次是合规需求（50%，仅对Web3用户）。

### 3.2 五大关键洞察

**洞察1：多供应商管理是隐形成本陷阱**
- 问题：开发者平均使用3-5个LLM提供商，每个都需要独立SDK、密钥管理、账单对账
- 解决方案：渐进式复杂度功能设计
  - Phase 1：仅提供路由层聚合（快速接入）
  - Phase 2：开放自建推理层（成本优化）
  - Phase 3：开放$CRED代币体系（财务透明）

**洞察2：开发者需要路由可解释性**
- 问题：黑盒路由导致开发者不信任，无法排查问题
- 解决方案：提供路由决策日志
  - 每条请求附带路由理由（如"复杂度评分6/10，路由至DeepSeek-V4自建集群"）
  - 提供路由历史查询和可视化分析
  - 支持自定义路由规则（专家模式）

**洞察3：Web3计费需渐进式信任构建**
- 问题：开发者对纯链上计费有安全顾虑，担心gas费波动
- 解决方案：混合模式
  - 入口：支持法币（Stripe/支付宝）和加密货币（USDT/$CRED）
  - 计费：链下Redis实时计费，链上L2每日结算
  - 透明度：可选公开链上账本供审计

**洞察4：SaaS企业决策链决定产品必须财务友好**
- 问题：SaaS企业采购需要多层审批，财务/CTO/CFO关注点不同
- 解决方案：财务级报表套件
  - 多维度报表：按项目/团队/用户/模型/时间
  - 导出格式：CSV/PDF/Excel
  - 告警机制：预算消耗达80%/100%时通知

**洞察5：接入体验是最后一公里流失点**
- 问题：迁移成本高，开发者倾向于"能用就不换"
- 解决方案：零代码改动迁移
  - OpenAI-Compatible API：只需改base_url
  - SDK自动适配：Python/Node/Go多语言SDK
  - 灰度回滚：支持按比例切流量，秒级回滚

### 3.3 痛点优先级

| 优先级 | 痛点 | 影响用户比例 | 当前解决方案 |
|-------|------|------------|-------------|
| P0 | 多供应商SDK不一致 | 85% | 手动封装适配器 |
| P0 | 计费碎片化 | 80% | 每月手动汇总账单 |
| P0 | 缺乏实时成本可见性 | 75% | 无法实时查看 |
| P0 | 故障时无自动兜底 | 70% | 手动切换，响应慢 |
| P1 | 路由策略固化 | 60% | 无法自定义 |
| P1 | 缓存效率低 | 55% | 无缓存或自建简单缓存 |

### 4.3 SWOT 分析

**优势（Strengths）**
- 全球唯一"智能路由+自建模型+Web3对账"三位一体平台
- Judge Agent智能路由降本40-85%（经数析验证）
- 自建DeepSeek-V4集群，成本优势显著
- $CRED区块链对账，透明度行业领先

**劣势（Weaknesses）**
- 早期阶段，用户基础为零
- 技术栈复杂（LLM路由+vLLM+区块链）
- 品牌信任度需要时间建立
- 团队规模有限，资源紧张

**机会（Opportunities）**
- LLM AI网关市场CAGR 45.8%，市场窗口期3-5年
- 75%+团队使用多模型，但仅13%实现企业级路由
- Web3计费接受度逐步提升
- 中小企业SaaS需求旺盛

**威胁（Threats）**
- OpenRouter可能推出智能路由功能
- 云厂商（阿里云/AWS）可能推出类似服务
- 去中心化算力（Akash）可能扩展API网关能力
- 经济下行可能压缩企业IT预算

### 4.4 竞争壁垒

1. **技术壁垒**：Judge Agent需要大量标注数据+持续优化，数据飞轮效应明显
2. **规模壁垒**：日均处理≥50M tokens才能实现盈亏平衡，后来者难以追赶
3. **生态壁垒**：$CRED代币经济形成网络效应，持有者越多越有价值
4. **合规壁垒**：Web3计费需持牌，合规门槛形成天然护城河

| 8 | **不支持本地部署的模型** | 仅支持已接入的自建集群和商业API |
