# MaaS-Router 产品路线图更新方案

**日期**：2026-05-05
**类型**：路线图更新
**参与成员**：路径（路线图规划师）、瑞思（用户研究员）、竞析（竞品分析师）、数析（数据分析师）、析客（需求分析师）

---

## 📌 TL;DR（执行摘要）

- **核心目标**：18 个月内从 MVP 推进至去中心化推理网络试点，以"智能路由降本 + 自建模型托管 + Web3 透明对账"三位一体差异化切入 LLM AI 网关市场（CAGR 45.8%），12 个月目标 QPS 2,000、成本节省 55%、月活开发者 20,000。
- **关键决策**：Phase 1-2 聚焦 AI 开发者（Primary Persona），以零代码迁移体验和实时成本可见性解决 P0 痛点；Phase 3 引入 $CRED 链下积分→链上结算的渐进式信任模型；Phase 5 开放生态时优先引入头部模型供应商建立网络效应。
- **风险焦点**：OpenRouter 已集成 Not Diamond Auto Router 是最大竞品威胁，需在 Month 3-4 尽早上线智能路由 Agent 建立先发优势；区块链监管变化需法务持续跟踪。
- **下一步**：本周启动 Phase 1 工程排期，确认 2 家商业 API 合作意向，完成自建 DS-V4 单节点部署架构评审。

| 指标 | Month 3 | Month 6 | Month 12 | 测量方式 |
|------|---------|---------|----------|----------|

| Latency P50 | 500ms | 300ms | 200ms | 分布式追踪（OpenTelemetry） |
| SLA | 99.5% | 99.9% | 99.95% | 可用性监控（月度） |

### 成本模型要点

- **60% 简单请求分流至自建层**为 12 个月核心目标
- **盈亏平衡线**：日均 200M+ tokens，自建层 2-4 个月回本
- **可行性验证**：日均 ≥50M tokens → 40% 节省；日均 ≥200M tokens → 60% 节省
- **关键假设**：自建 DS-V4 推理成本低于商业 API 的 40%（需 Phase 1 实测验证）

### 市场规模

- LLM AI 网关市场：2025 年 $1,840 万 → 2032 年 $2.5 亿（CAGR 45.8%）
- AI 推理市场：2025 年 $1,037 亿 → 2030 年 $2,550 亿（CAGR 19.2%）
- **切入逻辑**：AI 网关是推理市场的"入口层"，从网关切入可向上游（推理）和下游（应用）双向延伸

### Phase 1：MVP — 统一网关（Month 1-2）

**目标**：以"零代码迁移"体验解决 AI 开发者最痛的 P0 问题，验证统一网关 + 自建模型的基本价值假设。

**关键交付**：

| 交付物 | 说明 | 验收标准 |
|--------|------|----------|
| OpenAI 兼容 API | `POST /v1/chat/completions` 兼容格式，支持 stream | 100% OpenAI SDK 可直接调用 |
| 2 家商业 API 集成 | 初步建议：DeepSeek API + OpenAI（覆盖国产+国际） | 正常请求 + 自动 failover |
| 自建 DS-V4 单节点 | GPU 推理服务 + 健康检查 + 负载上报 | 单节点 QPS ≥20, P50 <500ms |
| 基础路由规则 | 基于模型名/请求长度的硬编码规则 | 简单请求命中率 ≥60% |
| 实时仪表盘 | 请求量、延迟、成本、错误率 Dashboard | 数据延迟 <10s |
| 5 分钟接入指南 | 文档 + 示例代码 + Playground | 新用户 5 分钟内完成首次调用 |

**成功指标**：
- QPS 稳定 50+（Day 1 可支撑）
- 新用户接入耗时 <5 分钟
- P0 痛点覆盖率 100%（4/4 痛点有对应功能）
- 自建层承担 ≥40% 简单请求

**资源需求**：4-6 名后端工程师 + 1 名前端 + 1 名 PM + 1 名设计

---

### Phase 2：智能路由 Agent + $CRED 链下积分（Month 3-4）

**目标**：上线智能路由核心能力，建立与竞品（尤其是 OpenRouter Not Diamond）的技术差异化；同时引入 $CRED 积分系统为 Phase 3 Web3 结算铺路。

**关键交付**：

| 交付物 | 说明 | 验收标准 |
|--------|------|----------|
| Qwen2.5-7B Judge Agent | 基于 prompt complexity 的路由决策模型 | 路由准确率 ≥85%（vs. 人工标注） |
| 路由决策日志 | 每次请求记录路由原因 + 置信度 | 可查询 + 可导出 |
| Playground | 在线测试界面，支持模型对比、成本预估 | 响应时间 <2s |
| $CRED 链下积分系统 | 请求消耗积分 + 积分余额查询 + 充值（法币） | 1 CRED = 1M tokens 基础定价 |
| Budget Alert | 预算阈值告警 + 请求限流 | 告警延迟 <30s |
| 路由规则配置 API | 支持用户自定义路由规则 | 支持模型名 / prompt 长度 / 标签维度 |

**成功指标**：
- QPS 100
- 路由准确率 85%（核心北极星指标）
- 成本节省 35%
- MAD 1,000
- Latency P50 500ms
- 路由决策日志覆盖率 100%

**资源需求**：+1-2 名 ML 工程师（路由模型训练与调优）+ 延续 Phase 1 团队

**风险与缓解**：
- **⚠️ 最高风险**：路由准确率不达标 → 准备 fallback：先用规则引擎兜底，Judge 仅作推荐
- *积分系统冷启动* → Phase 1 用户免费赠送 $CRED 体验额度
- *竞品 OpenRouter 加速* → 将 Playground 做成核心增长工具，强调"成本可视化"差异

---

### Phase 3：Web3 结算层（Month 5-6）

**目标**：将 $CRED 从链下积分升级为链上结算协议，实现多供应商自动分账的透明化对账。

**关键交付**：

| 交付物 | 说明 | 验收标准 |
|--------|------|----------|
| L2 每日 Merkle 结算 | 每日凌晨将当日账单生成 Merkle Tree 上链 | 结算延迟 <24h, 数据完整性可验证 |
| 100% 准备金证明 | $CRED 对应的法币/稳定币储备金链上公示 | 第三方可独立验证 |
| 多供应商自动分账 | 根据实际路由用量自动按比例分配收入 | 分账误差 <0.01% |
| 财务级报表 | 月度账单 + 对账单 + 发票模板 | 符合企业财务审计要求 |
| 法币 ↔ $CRED 兑换通道 | 用户可用法币充值 $CRED、供应商可提现 | KYC 合规、T+1 到账 |

**成功指标**：
- QPS 500
- SLA 99.9%
- 成本节省 45%
- MAD 5,000
- 供应商分账准确率 99.99%
- 准备金证明日更新

**资源需求**：+2 名区块链工程师（智能合约 + L2 集成）+ 1 名法务顾问（合规审查）

### Phase 4：企业版 + 多模态（Month 7-9）

**目标**：从开发者市场向上突破至 SaaS 企业客户，提升 ARPU；同时扩展多模态能力。

**关键交付**：

| 交付物 | 说明 | 验收标准 |
|--------|------|----------|
| VPC 私有化接入 | 企业可在自有 VPC 部署网关节点 | 数据不离开企业网络 |
| 企业 SLA | 99.9%+ SLA + 赔付条款 | 合同模板就绪 |
| 审计日志 | 全量请求日志 + 操作日志 + 导出 | SOC 2 Type II 准备 |
| RBAC 权限管理 | 团队成员角色 + API Key 分级 | 支持 Admin/Developer/Viewer |
| 图像路由 | 图像生成请求自动分发 | 路由准确率 ≥80% |
| 语音路由 | TTS/ASR 请求智能分发 | 路由准确率 ≥80% |
| SSO 集成 | SAML/OIDC 单点登录 | 支持 Okta/Azure AD |

**成功指标**：
- QPS 1,000
- Enterprise 客户签约 20+
- 多模态请求占比 ≥15%
- 企业客户 NPS ≥50

## 5. 利益相关者沟通要点

### 高管视图（Executive Summary）

**商业价值**：
- **市场规模**：切入 $2.5 亿 AI 网关市场（2032 年），CAGR 45.8%
- **收入模型**：三层收入——路由服务费（按 token 抽成）+ $CRED 结算服务费 + 企业版订阅费
- **12 个月目标**：20,000 月活开发者，目标 ARR $200-500 万

**投资回报**：
- Phase 1-2（4 个月）：投入 ~$50-80 万（人力+GPU），产出 MVP + 智能路由，验证 PMF
- 盈亏平衡点：日均 200M+ tokens（约 Phase 3 中期可达成）
- 自建 GPU 投资回收期：2-4 个月（达到 200M tokens 日均后）

### 工程视图（Engineering View）

**技术架构演进**：

```
Phase 1:   Client → API Gateway → [Rule Engine] → Provider A / Provider B / Self-hosted DS-V4
Phase 2:   Client → API Gateway → [Qwen2.5-7B Judge] → [Rule Fallback] → Providers
Phase 3:   + L2 Settlement Layer (Merkle Tree + $CRED Contracts)
Phase 4:   + VPC Peering + Multimodal Router + Audit Pipeline
Phase 5:   + Provider Marketplace + Smart Contract Escrow
Phase 6:   + P2P Inference Network + Node Incentive Layer
```

**资源需求总览**：

| 阶段 | 工程人力 | GPU 资源 | 基础设施 |
|------|---------|---------|---------|
| Phase 1-2 | 6-8 人 | 1-2×A100/H100 | 云部署 |
| Phase 3-4 | 10-12 人 | 4-8×A100/H100 | 混合部署 |
| Phase 5-6 | 12-15 人 | 8-16×A100/H100 | 边缘节点 |

### 设计视图（Design View）

**用户体验核心原则**：
1. **零摩擦接入**：OpenAI SDK 兼容，5 分钟完成首次调用
2. **成本透明**：每次请求展示实际成本 + 路由决策依据
3. **渐进式复杂度**：默认开箱即用，高级功能按需开启
4. **开发者审美**：深色主题、CLI 友好、API-First 设计
