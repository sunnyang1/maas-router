## 核心组件说明

### 1. API Gateway (后端网关)

**职责：**
- 接收所有 API 请求
- 执行认证和授权
- 限流和熔断
- 请求路由和负载均衡
- 响应格式化和错误处理

**技术栈：**
- Go 1.22+
- Gin Web Framework
- Wire 依赖注入

**核心模块：**

| 模块 | 功能 | 说明 |
|------|------|------|
| Auth Middleware | JWT/API Key 认证 | 支持双认证模式 |
| Rate Limiter | 请求限流 | 基于 Redis 的滑动窗口算法 |
| Router Engine | 智能路由决策 | 集成 Judge Agent |
| Billing Engine | 实时计费 | 精确到 token 级别的计费 |
| Provider Manager | 供应商管理 | 健康检查和故障转移 |

### 2. Judge Agent (智能路由 Agent)

**职责：**
- 分析请求复杂度
- 预测推理成本
- 做出路由决策

**技术栈：**
- Python 3.11
- FastAPI
- Transformers (Qwen2.5-7B)
- vLLM (推理加速)

**评分维度：**

```python
class ComplexityScore:
    """复杂度评分模型"""
    
    # 输入特征
    input_tokens: int          # 输入 token 数量
    context_complexity: float  # 上下文复杂度 (0-1)
    task_type: TaskType        # 任务类型分类
    
    # 输出预测
    expected_output_tokens: int    # 预期输出长度
    reasoning_depth: float         # 推理深度 (0-1)
    
    # 综合评分 (0-1)
    final_score: float
```

**路由策略：**

| 评分区间 | 复杂度 | 推荐路由 | 理由 |
|----------|--------|----------|------|
| 0.0 - 0.3 | 简单 | DeepSeek-V4 | 成本低，响应快 |
| 0.3 - 0.6 | 中等 | DeepSeek-V4 | 平衡成本和质量 |
| 0.6 - 0.8 | 较难 | GPT-3.5-Turbo | 质量优先 |
| 0.8 - 1.0 | 复杂 | GPT-4 | 确保输出质量 |

### 3. Provider Manager (供应商管理器)

**职责：**
- 管理多个 AI 供应商
- 健康检查和故障检测
- 负载均衡和故障转移
- 成本优化

**支持的供应商类型：**

```go
type ProviderType string

const (
    ProviderTypeSelfHosted ProviderType = "self_hosted"  // 自建集群
    ProviderTypeCommercial ProviderType = "commercial"  // 商业 API
    ProviderTypeHybrid     ProviderType = "hybrid"      // 混合模式
)
```

**故障转移策略：**

```
1. 主供应商健康检查 (每 5 秒)
   ↓ 失败
2. 自动切换到备用供应商 (< 30 秒)
   ↓ 恢复
3. 渐进式流量回切 (防止震荡)
```

### 4. Billing Engine (计费引擎)

**职责：**
- 实时 token 计数
- 成本计算
- 配额管理
- 结算队列

**计费模型：**

```go
type BillingModel struct {
```

### 1. 正常请求处理流程

```
┌────────┐     ┌────────────┐     ┌────────────┐     ┌────────────┐     ┌────────┐
│ Client │────▶│   API      │────▶│   Judge    │────▶│  Provider  │────▶│ Client │
│        │     │   Gateway  │     │   Agent    │     │            │     │        │
└────────┘     └────────────┘     └────────────┘     └────────────┘     └────────┘
                    │                   │                   │
                    │                   │                   │
                    ▼                   ▼                   ▼
               ┌────────────┐     ┌────────────┐     ┌────────────┐
               │  Auth/     │     │  Complexity│     │  Response  │
               │  Rate      │     │  Analysis  │     │  Stream    │
               │  Limit     │     │  (100ms)   │     │            │
               └────────────┘     └────────────┘     └────────────┘
```

**详细步骤：**

1. **请求接收** (API Gateway)
   - 解析 HTTP 请求
   - 验证 API Key
   - 检查限流配额

2. **复杂度分析** (Judge Agent, ~100ms)
   - 提取请求特征
   - Qwen2.5-7B 推理评分
   - 返回复杂度分数和置信度

3. **路由决策**
   - 根据评分选择供应商
   - 考虑成本和可用性
   - 执行路由策略

4. **请求转发**
   - 转换请求格式
   - 添加供应商认证信息
   - 流式转发响应

5. **计费记录**
   - 实时统计 token 使用量
   - 计算费用
   - 写入结算队列

## 技术选型理由

### 后端：Go + Gin

**选择理由：**
- **高性能**：Go 的 goroutine 模型适合高并发 API 网关
- **低延迟**：编译型语言，启动快，内存占用低
- **生态成熟**：丰富的中间件和工具链
- **部署简单**：静态编译，单二进制文件

**对比其他方案：**

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| Node.js | 生态丰富 | 单线程，CPU 密集型任务性能差 | 不适合 |
| Python | 开发快 | GIL 限制，性能不足 | 仅用于 Judge Agent |
| Java | 生态完善 | 启动慢，内存占用大 | 过于重量级 |
| Rust | 性能极致 | 学习曲线陡峭，开发效率低 | 过度设计 |

### ORM：Ent

**选择理由：**
- **类型安全**：编译时检查，避免运行时错误
- **代码生成**：自动生成类型安全的查询代码
- **图遍历**：支持复杂的关系查询
- **迁移工具**：内置数据库迁移支持

### 前端：Next.js + React

**选择理由：**
- **全栈能力**：SSR/SSG 支持，SEO 友好
- **性能优化**：自动代码分割、图片优化
- **开发体验**：热更新、TypeScript 原生支持
- **部署灵活**：支持多种部署模式

### 智能路由：Python + FastAPI

**选择理由：**
- **ML 生态**：PyTorch/Transformers 生态完善
- **异步性能**：FastAPI 基于 Starlette，性能优秀
- **开发效率**：Python 在 ML 领域开发效率高
- **模型服务**：vLLM 等推理框架原生支持 Python

### 数据库：PostgreSQL + Redis

**选择理由：**
- **PostgreSQL**：
  - ACID 保证，数据一致性
  - JSONB 支持，灵活存储
  - 强大的查询优化器
  - 成熟的主从复制方案

- **Redis**：
  - 内存存储，极低延迟
  - 丰富的数据结构
  - 内置限流、分布式锁实现

### 区块链：Solidity + Polygon

**选择理由：**
- **Polygon**：
  - 低 Gas 费用 (~$0.001/tx)
  - 快速确认 (~2s)
  - EVM 兼容，生态丰富
  - 与以太坊的安全桥接

## 安全设计

### 认证与授权

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Client    │────▶│   API Key   │────▶│   JWT       │
│   Request   │     │   Validate  │     │   Issue     │
└─────────────┘     └─────────────┘     └─────────────┘
                                               │
                                               ▼
                                        ┌─────────────┐
                                        │   RBAC      │
                                        │   Check     │
                                        └─────────────┘
```

**安全机制：**

| 层级 | 机制 | 说明 |
|------|------|------|
| 传输层 | TLS 1.3 | 强制 HTTPS |
| 认证层 | API Key + JWT | 双因素认证 |
| 授权层 | RBAC | 基于角色的权限控制 |
| 应用层 | 签名验证 | 请求签名防篡改 |

### 数据安全

- **敏感数据加密**：数据库中 API Key、私钥等使用 AES-256 加密
- **传输加密**：所有服务间通信使用 mTLS
- **审计日志**：所有敏感操作记录审计日志
- **数据脱敏**：日志中敏感信息自动脱敏

### 智能合约安全

- **多重签名**：关键操作需要多签
- **时间锁**：大额转账延迟执行
- **审计**：合约代码通过第三方审计
- **保险基金**：设立保险基金应对极端情况
